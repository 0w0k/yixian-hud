## 0. 两个项目是什么关系？

| | **yixian-card-counter-with-proxy**（根基/上游） | **yixian-hud**（衍生/下游，你现在这个） |
|---|---|---|
| 仓库 | `Airexplosion/yixian-card-counter` | `0w0k/yixian-hud` |
| 形态 | **外部窗口**（pywebview 置顶小窗，游戏旁边） | **游戏内注入 HUD**（frida 注入，画面上叠字） |
| 独有 | 完整 `game-api`（控制游戏）+ `autoplay`（自动打牌）+ `overlay` | `native_hud/csharp`（注入用的 C# 层）+ HUD 渲染 |
| 共享 | **`proxy/` 解码层、`proxy_view.py` 视图模型、`web/yisim` 伤害引擎** ←—— 两边几乎一样 | 同左 |

一句话：**两个项目共用同一套"读网络包→解码→视图模型"的大脑，只是"显示/操作的躯壳"不同**。
- 根基项目把结果画在**外部小窗**，还能反过来**操作游戏**（点按钮、自动摆牌）。
- 你这个 HUD 把结果**注入进游戏画面**叠加显示。

学习时：**解码/数据这块两边读哪个都行**（建议读根基项目，它样例数据全）；**显示/注入**各看各的。

## 1. 核心思想：辅助工具怎么"看懂"游戏？

**不截图、不识图**。游戏和服务器之间走 WebSocket 通信，我们用**中间人代理（mitmproxy）**把这些网络包截下来解码，于是对局的真实状态（你的手牌、对手的板、命/修为…）**直接读出来，分毫不差**。

```
游戏 ⇄ WebSocket(加密) ⇄ mitmproxy代理(我们的addon拦截) → 解码 → GameState 对象
                                                                  │
                                                          proxy_view 组装成 view-model(JSON)
                                                                  │
                                          ┌───────────────────────┴───────────────────────┐
                                  根基:推给 pywebview 窗口                  HUD:推给注入的 C# 叠加层
```

要"看懂游戏"= 看懂这条链路。下面从最原始的字节，一层层往上剥。

## 2. 由浅入深的学习路径（7 个阶段）

### 阶段 0 — 先跑起来，产出样例
根基项目设了 `YX_CAPTURE=1` 就会把每帧落盘。你不一定要自己跑——**根基项目里已经有现成的真实样例数据**（见第 3 节），直接拿来读最快。

### 阶段 1 — 看懂"一帧"长啥样（最外层）
打开 `proxy/output/traffic.jsonl`，每行是一个 WebSocket 帧：
```json
{"type":"ws_frame", "direction":"server->client",
 "url":"https://p3.hk.darksungame.com/game-server/.../...",
 "decoded":{"msgpack":[...], "strings":[...]}}
```
**关注 `direction`**：`server->client`=游戏收到的（读状态），`client->server`=你的操作。

### 阶段 2 — 剥开"信封"，找到业务消息（`proxy/decoder.py`）
`decoded.msgpack` 是个数组，**第 2 个元素就是业务消息信封**：
```json
["data", {"type":"GameStatus", "data":"CAEQH..."}]
         └─ type=消息名      └─ data=base64 的 protobuf 字节
```
- `type` 告诉你这是什么消息（`GameStatus`/`MoveCardReq`/…）。
- `data` 是内层 protobuf，需要再解一层。
- 解析这层的代码：`addon._inner(mp)` 就是取 `mp[1]`。

### 阶段 3 — 解开 protobuf，认字段（核心难点）
内层 `data` 用 `blackboxprotobuf` **无 schema 解码**，解出来字段是**纯数字号**（如 `200.6`），语义全靠**流量分析逆出来**。这是逆向最硬核的部分。真实解码长这样（`proxy/output/gs_field_dump.txt`）：
```
## player[0] · id=639410db...
  103: list[12] = [{'1':6000002,'2':2001}, {'1':6000010,'2':3001}, ...]   ← 持有的牌
  3: 100                                                                   ← 命(destiny)
  2: bytes = e7ad96...                                                     ← 中文名(utf-8)
  13: bytes[8] = 7d5984015bb9ea01                                         ← 仙命(packed varint)
  200: {
    3: 65    ← 修为(xiuwei)
    4: 5     ← 境界(realm 1..5)
    6: bytes ← 摆牌(8个packed varint)
    8: {2: 8}← 体魄(tipo) 藏在 8.2
  }
```
**怎么逆出"200.4=境界"的？** 对比多帧：突破时这个值 +1、范围恒在 1~5 → 推断是境界。逆向就是"控制变量法看哪个数字随哪个游戏行为变"。
字段映射的权威表在 `game_state.py` 文件头注释 + dataclass 定义。

### 阶段 4 — 解开"打包数组"（varint 列表）
`13`（仙命）、`200.6`（摆牌）这种是 **packed varint**（一串连续的变长整数）。`shadow_state.decode_varint_list` 负责解。真实例子（已验证，见第 3 节 demo）：
```
仙命字节 7d5984015bb9ea01 → [125, 89, 132, 91, 30009]
                          → ['猛虎之躯','木灵炼体','锟铻金环','刚劲','云巅雷劫']
```
解出 id 后再查字典（`fate_id_map.json` / `card_id_map.json`）得名字。

### 阶段 5 — 影子状态：比服务器更早知道你的操作（`shadow_state.py`）
只读服务器回包有延迟。所以我们**也拦截你发出的操作**（`client->server` 的 `MoveCardReq`/`RefineCardReq`…），在本地**增量重放**出一份"影子手牌/板"，比服务器确认更早更新 UI。`shadow_log.txt` 是它的人类可读日志：
```
[17:35:35] MoveCardReq {3:1} [0:0]→[1:0] → PLACE board[0] 斗转星移
                hand:  []
                board: ['斗转星移','_','_','_','_','_','_','_']
```

### 阶段 6 — 控制面：反过来操作游戏（仅根基项目，frida 注入）
读懂还不够，根基项目还能**下命令**：frida 注入 `YiXianApi.dll`，Python 调游戏内 C# 方法。例如 `api.battle.skip()` 跳过战斗、`api.placement.move(i,j)` 摆牌。声明表在 `native_hud/api/registry.py`，明细见 `docs/api/game-reverse-engineering.md` B 节。

### 阶段 7 — 做你自己的分析辅助
**好消息：做分析你基本不用碰阶段 2~5。** proxy_view 已经把一切解码成干净的 `view-model`（见 demo 6），你只要消费它。见第 4 节。

## 4. 你要做"分析辅助"，从哪下手？

**结论：消费 `view-model` 就够了，不用碰解码层。** proxy_view 每帧推给你一个上面 Demo 6 的 JSON，里面有双方的板/手牌/命/修为/仙命/牌库剩余。你的分析逻辑只是这个 JSON 的**纯函数**。

三个由易到难的练手切入点：

1. **统计型（最易）**：基于 `me.hand[].copiesLeftInDeck` 和卡名做"关键牌还剩几张/抽中概率"，或"危险牌预警"（对手板上出现某些牌就提示）。参考现成的 `web/recommend.js`、`proxy_view.Counter`。

2. **模拟型（中）**：把 `me.board`/`opponent.board`/仙命喂给 `web/yisim`（伤害引擎），算战局走势。入参格式见 `docs/yisim-damage-params.md`（你上一轮刚改过的铸剑师层数就在这层）。

3. **决策型（难）**：结合 view-model + yisim 给"该摆哪张/该换哪张"的建议，甚至接根基项目的 `game-api` 自动执行（`autoplay/`）。

**接线位置**：
- HUD（A）：`native_hud/bridge/hud_launcher.py` 的各 `*_loop` 消费 `_latest["vm"]`，算完调 C# 面板显示。你上一轮加的 `[total-in]` 日志就在 `total_loop`。
- 根基（B）：`proxy_view.py` 产出 vm → `web/ui.js` 渲染。

## 5. 关键文件地图

| 层 | 文件 | 干嘛的 |
|---|---|---|
| 拦截 | `proxy/addon.py` | mitmproxy addon，按消息 `type` 分发到各 `_handle_*` |
| 解码 | `proxy/decoder.py` | Colyseus/msgpack/protobuf 三层拆包 |
| 字段 | `proxy/game_state.py` | protobuf 数字字段 → `GameState`/`PlayerState` dataclass（**字段映射权威**） |
| 影子 | `proxy/shadow_state.py` | 增量重放我方操作；varint 解码；季节区/突破/灵羽合并 |
| 字典 | `proxy/*_id_map.json` | id→名（卡2915/仙命496/职业7）、仙命模拟元数据412 |
| 视图 | `proxy_view.py` | `GameState` → 干净 JSON `view-model`（**分析从这消费**） |
| 模拟 | `web/yisim.bundle.js` | 伤害/战局模拟引擎 |
| 显示A | `native_hud/bridge/hud_launcher.py` + `csharp/` | 注入 + HUD 叠加 |
| 显示B | `web/ui.js` | pywebview 窗口渲染 |
| 控制B | `native_hud/api/registry.py` | game-api 声明表（操作游戏） |

## 6. 初学者避坑

- **字段号会变**：游戏更新可能改 protobuf 字段含义（代码注释里 `R24/R28` 就是"第几次逆向修订"）。看到数字对不上先怀疑游戏版本变了，用 `YX_CAPTURE=1` 抓新样例对比。
- **HP 不在网络上**：真实血量靠读本地 `BattleLog.json`，纯网络解码拿不到（见接口文档 C 节缺口）。
- **我方 vs 对手数据来源不对称**：对手数据来自 GameStatus 全量广播，我方有些走增量监听——你上一轮修的"我方仙命空"就是这个坑（中途接代理会漏增量）。优先用全量广播的字段。
- **schema 帧解不了**：Colyseus `0x0e/0x0f` 是另一种二进制编码，当前记为 `ws_undecoded`，想啃这块是进阶课题。
- **改解码会影响两个项目**：`proxy/` 是共享大脑，在 A 改了同样的逻辑 B 也得同步（反之亦然）。

## 7. 建议的上手顺序

1. 读本文 + `docs/api/game-reverse-engineering.md`（建立全局图）。
2. 打开根基项目 3 个真实样例对着 Demo 看：`gs_field_dump.txt`（字段）→ `shadow_log.txt`（事件）→ `deck_tracker.jsonl`（view-model）。
3. 读 `game_state.py`（200 行，字段映射）+ `proxy_view.py` 的 `build_view_model`（看 vm 怎么拼）。
4. 挑第 4 节的"统计型"练手，在 `hud_launcher.py` 加一个 `*_loop` 消费 vm 输出你的分析。
5. 想更深再啃 `shadow_state.py`（影子重放）和 `addon.py`（分发）。
